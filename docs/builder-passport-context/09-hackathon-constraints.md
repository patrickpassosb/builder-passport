# Hackathon Constraints

These constraints were derived from the event materials and should guide implementation.

## Rules
- The project must be new and conceived specifically for Monad Blitz
- Do not continue an existing personal project
- Do not fork an existing codebase except standard libraries and boilerplates
- Research and planning before the event are encouraged
- The project repository must be public
- The project must be deployed and working on Monad Testnet
- The event values experimentation and innovation more than polish

## Practical implications
- Use a boilerplate if needed, but do not submit an old project
- Keep the project clearly original in idea and implementation
- Optimize for a strong demo, not maximum feature count
- Do not spend the day on infrastructure that is unrelated to the core idea

## Operational notes from pre-event discussion
- Going solo is allowed
- Participants should bring their own laptops
- It was recommended to arrive with groups or at least a clear concept because the event is fast-shipping oriented
- Pre-event brainstorming is encouraged because ideation time on the day is expensive

## Product rule for this project
Do not try to build a full Devpost/DoraHacks competitor.
This project is a reputation layer for hackathons.
